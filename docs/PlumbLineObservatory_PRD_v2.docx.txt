1. Overview
Plumb Line Observatory is an interactive 3D data visualisation tool that maps manipulation patterns and biblical principles in modern media, political, and religious discourse.
The project takes its name from the prophetic image of the plumb line in Amos 7:7-8 — a builder's tool for measuring straightness, used by God to test whether Israel's moral and social structures were true. In the same way, the Observatory holds a straight line against modern discourse to ask: what techniques are being used, what frameworks are being invoked, and do they align across ideological boundaries we assume to be different?
Built on React 19 and Three.js, the tool renders 13 discourse analysis cases (nodes) as glowing 3D spheres in a star-field environment, coloured by ideological tribe: Right, Left, Religion, and Geopolitics. Each node is scored across seven manipulation dimensions and seven biblical principles, producing a five-axis "plumb score" (process, locus, uplift, means, blueprint) that functions as a structural fingerprint.
Phase 2 of the project introduces multiple view modes — a 2D force-directed graph, timeline scrubber, and fingerprint radar view — transforming the Observatory from a beautiful artefact into a multi-lens epistemology instrument.


2. Problem Statement
2.1 The Core Epistemic Problem
Modern audiences consume discourse through tribal filters. A reader of The Guardian and a reader of The Daily Telegraph exist in structurally separate epistemic worlds — different sources, different editorial angles, different interpretive frameworks. What is invisible to both is that the rhetorical machinery underneath their respective sources is often identical. Both may use urgency framing. Both may use asymmetric attribution. Both may invoke structural justice frameworks derived from the same biblical tradition.
The result is a population that argues passionately about conclusions while sharing no common ground for evaluating the quality of the reasoning that produced those conclusions. This is not merely a political problem — it is a formation problem. People cannot develop epistemic integrity if they cannot see the structure of what they are consuming.


  i  
	The Plumb Line Observatory was designed to surface this invisible structural layer — to show that ideological enemies often use the same weapons, and that the weapons predate the fights.
	

2.2 The Design Gap
Existing fact-checking tools operate at the claims level (true/false). Media bias trackers operate at the source level (left/right). Neither operates at the rhetorical structure level. The Observatory is the only tool that cross-maps manipulation patterns and theological frameworks simultaneously, revealing connections across ideological lines that no single-axis analysis could surface.
2.3 What Phase 1 Solved
Phase 1 (live) delivered the 3D sphere visualisation with 13 hand-crafted nodes. It proved the concept, established the data model, and produced a genuinely stunning visual artefact. What it could not do:
* Show the force relationships between nodes clearly — the 3D sphere obscures clustering
* Communicate change over time — there is no temporal dimension
* Let users interrogate a single node's structural fingerprint at a glance — the detail panel is text-heavy
* Scale beyond ~20 nodes without the sphere becoming visually unreadable

5.4 Edge Logic
Edges between nodes are computed at render time — two nodes are connected if they share at least one manipulation type key or at least one biblical principle key. Edge weight is the count of shared keys. In Phase 2, edge weight drives visual thickness in the force graph and chord diagram views.
  !  
	Adding a date field to all existing nodes is a Phase 2 prerequisite. Without it, the timeline scrubber cannot function. All 13 existing nodes must be backdated before the timeline view is released.

6. Phase 2 Feature Specifications
Phase 2 introduces four distinct view modes, a global view-switcher UI, and per-node enhancements. Each mode surfaces different structural truths about the data.


6.1 View Mode Switcher
A persistent top-of-screen mode bar replaces the current absent navigation. The switcher is always visible, allows instant transition between all view modes, and persists the selected node/filter state across transitions.

6.2 2D Force-Directed Network (Graph Mode)
6.2.1 Purpose
The 3D sphere is visually spectacular but structurally ambiguous — nodes positioned on a sphere surface give no clustering information. The 2D force graph makes structural relationships the primary output: heavily connected nodes cluster together; isolated nodes float to the periphery. Cross-tribe clusters become impossible to miss.
6.2.2 Behaviour
* D3 force simulation with: charge repulsion (−300), link attraction (distance proportional to 1/edge_weight), centre gravity (pull toward viewport centre), collision detection (radius = node_radius + 12px)
* Nodes are circles, radius proportional to total edge count (min 12px, max 36px)
* Node colour follows the existing tribe colour system (Right = coral, Left = blue, Religion = gold, Geopolitics = purple)
* Edge lines are drawn with opacity and stroke-width proportional to edge weight (shared keys count)
* Nodes are draggable — dragging one node detaches it from the simulation temporarily, then releases back
* Clicking a node opens the existing right-panel detail view
* Hover highlights the node and dims all non-adjacent nodes/edges (focus mode)
* A "cluster by tribe" toggle groups nodes into four quadrants using a quadrant force, overriding the free simulation
* A "show labels" toggle renders node title beneath each circle (default: off above 30 nodes)
6.2.3 Acceptance Criteria
* Simulation reaches stable state within 3 seconds on first render
* Dragging any node does not drop below 30fps
* Cross-tribe shared-manipulation clusters are visually apparent without any filter active
* Cluster-by-tribe toggle transitions smoothly (500ms)
* Mode transition from 3D sphere to force graph completes in under 400ms


6.3 Fingerprint Radar View (Plumb Fingerprint Mode)
6.3.1 Purpose
The five plumb dimensions (process, locus, uplift, means, blueprint) are currently only readable as numbers in the node detail panel. The radar view makes them instantly legible as a visual signature — a "fingerprint" that is unique to each node and comparable across nodes. The most powerful use is overlaying two nodes from different tribes to show structural similarity or difference.
6.3.2 Behaviour
* Primary view: single-node radar chart — filled polygon on a five-axis grid, 1–5 scale
* Each axis is labelled (process, locus, uplift, means, blueprint) with a brief descriptor on hover
* The filled area uses the node's tribe colour at 40% opacity; the stroke is solid tribe colour
* Score values are shown at each vertex as a small label (e.g. "4 / 5")
* Comparison mode: a second node can be selected from a dropdown — its radar overlays in a contrasting colour (30% opacity fill) to reveal convergence and divergence
* The grid background shows a reference polygon at score 3.0 (midpoint) as a thin dashed line
* Below the chart: a manipulation tag cloud (bubble chart) showing all manipulation types present in the selected node(s), sized by frequency across all nodes with that type
* A "what does this mean?" expand section provides plain-English interpretation of the node's plumb profile
6.3.3 Acceptance Criteria
* Selecting any node instantly renders its radar (< 100ms)
* Overlay mode clearly distinguishes two nodes with colour and opacity
* On tablet viewports (768px+) the radar and comparison panel both fit without horizontal scroll
* Each axis label is readable without hover on desktop
* Radar view is navigable by keyboard (tab through nodes to change selection)


6.4 Timeline Scrubber (Timeline Mode)
6.4.1 Purpose
Discourse does not happen in a vacuum — it happens in a sequence, and patterns emerge over time. The timeline view shows when each analysis case was published or broadcast, enabling users to see whether certain manipulation techniques cluster around specific events (elections, crises, cultural flashpoints) and whether the biblical principles invoked shift over time.
6.4.2 Behaviour
* Horizontal SVG/Canvas timeline with nodes placed at their date position
* Y-axis: tribe swim lanes (Right, Left, Religion, Geopolitics) — nodes sit in their tribe row
* A draggable range scrubber at the bottom sets a visible time window (e.g. show only 2022–2024)
* Nodes outside the scrubber window are dimmed (10% opacity), not hidden
* A "play" button animates the scrubber from earliest to latest date at 1x, 2x, 4x speed
* Clicking a node at any position opens the standard detail panel
* Edge lines between nodes that share manipulation types are shown as curved arcs, visible when both nodes are in the active window
* A "highlight manipulation type" dropdown filters the arc display to only show edges from a specific pattern (e.g. show only URGENCY connections)
* The visible date range label updates in real time as the scrubber moves
6.4.3 Prerequisites
  !  
	All 13 existing nodes must have a date field added before the timeline view renders correctly. Node dates should reflect the publication or broadcast date of the analysed discourse, not the date of analysis. This is a data entry task estimated at 2 hours.
	6.4.4 Acceptance Criteria
* Timeline renders all 13 nodes in correct date order with no overlapping node circles
* Play animation runs at consistent speed across the full date range
* Scrubber drag is smooth at 60fps
* Edge arcs only display between nodes within the active time window
* Date range label is always accurate to the scrubber position
